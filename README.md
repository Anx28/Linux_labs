# Linux_labs

Коллекция Docker-лабораторий для практики Linux/DevOps CLI.

Каждая лаба — отдельная папка с самодостаточным Docker-окружением, набором задач и автопроверкой. Цель — не "прочитать про команды", а наработать **мышечную память** в формате "руки в терминале".

## Общий формат каждой лабы

Все лабы построены по одному шаблону, чтобы переключение между ними не требовало запоминания "а как тут запускается":

```
labsN_<тема>/
├── Dockerfile        # ubuntu:24.04 + копирование скриптов
├── generate.sh       # создаёт тестовую файловую структуру
├── tasks.md          # 7–10 заданий от easy к medium
├── answer.sh         # сюда пишется решение
├── solutions.sh      # эталонные ответы
├── check.sh          # автопроверка: ./check.sh <N>
└── README.md         # описание конкретной лабы
```

**Цикл работы внутри любой лабы:**

```
1. ./generate.sh        # сбросить и пересоздать тестовые данные
2. cat tasks.md         # прочитать задание
3. nano answer.sh       # написать своё решение
4. ./check.sh N         # автопроверка
5. FAIL → исправить, OK → следующая
6. застрял → cat solutions.sh
```

## Быстрый старт

Без клонирования (если образ уже в DockerHub):

```bash
docker run -it --rm vladimir7/find-lab
```

```bash
git clone https://github.com/anx28/Linux_labs.git
cd Linux_labs/labs1_find
docker build -t find-lab .
docker run -it --rm find-lab
```

## Зачем это всё

Конкретные команды (`find`, `grep`, `awk`, `systemctl`) — это базовый инструментарий сисадмина/DevOps. Чтение шпаргалок даёт иллюзию знания, но в реальной задаче в 3 ночи на проде вспоминается только то, что руки набирали десятки раз. Эти лабы — попытка перевести "прочитал" в "набил".

Формат изолированного Docker-окружения нужен по двум причинам:
1. Безопасно — никакой риск что-то сломать в основной системе командой типа `find / -mtime +7 -delete`.
2. Воспроизводимо — у каждого запуска одинаковая стартовая файловая структура, что делает автопроверку возможной.
